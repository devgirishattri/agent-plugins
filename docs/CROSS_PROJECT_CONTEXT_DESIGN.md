# Cross-Project Context Sharing

**Date**: 2026-03-29
**Updated**: 2026-06-10
**Status**: Implemented (project-local storage, version history, diffing, staleness warning); automatic dispatch attachment is future work

## Overview

The `session-context` plugin lets agent sessions working in different project directories exchange project knowledge. A session can generate a concise context snapshot, list available snapshots, load one into its current work, diff it against earlier versions, or share it with another named tmux session.

This is useful when an orchestrator session dispatches work to another project. The receiving session can load API contracts, schemas, conventions, and architecture notes before making changes.

## Commands

| Command | Purpose |
|---------|---------|
| `/context-generate [snapshot-name]` | Generate a context snapshot for the current project (overwriting archives the previous version) |
| `/context-list` | List available snapshots (name, lines, last updated, history version count) |
| `/context-load <snapshot-name>` | Load a snapshot into the current session (warns when stale) |
| `/context-diff <snapshot-name> [--versions \| <timestamp>]` | Diff a snapshot against an archived version, or list archived versions |
| `/context-share <session> [snapshot-name]` | Notify another named session about a snapshot |
| `/context-remove <snapshot-name>` | Delete a snapshot |

## Snapshot Storage (project-local)

Snapshots are **project-local**, stored inside the repository being worked on:

```text
<git-root>/tmp/contexts/<name>.md                                # current snapshot
<git-root>/tmp/contexts/.history/<name>.<YYYYMMDD-HHMMSSZ>.md    # archived versions (UTC)
```

The root resolves via `git rev-parse --show-toplevel`, falling back to the current directory outside a repo. Both runtime implementations resolve the same directory, so a snapshot generated by one runtime is immediately loadable by the other.

The repo contains separate provider implementations of the same scripts:

| Provider | Plugin Path |
|----------|-------------|
| Claude | `plugins/session-context/` |
| Codex | `codex/plugins/session-context/` |

### Superseded design: provider-local storage

An earlier revision of this document specified provider-local snapshot directories (`~/.claude/context-snapshots/` for Claude, `~/.codex/context-snapshots/` for Codex) so the runtimes would not consume each other's data. **That design is superseded** and was never how the shipped scripts behave.

Rationale for project-local storage:

- **The shared project directory is the trust and sharing boundary.** A snapshot describes one project; co-locating it with that project scopes it correctly and keeps it out of unrelated work.
- **Both runtimes read the same snapshots.** Cross-runtime handoff (a Claude pane generating, a Codex pane loading, or vice versa) only works if both resolve the identical path — provider-local homes would silo the data.
- **Downstream features depend on it.** `/context-share` deliberately does not copy files — it assumes the recipient pane in the same repo already sees `tmp/contexts/`. The scheduler's `--context` attach on dispatch likewise resolves snapshots per-project.

## Version History and Diffing

When a save would overwrite an existing `<name>.md`, the old file is first copied to `tmp/contexts/.history/<name>.<UTC timestamp>.md`. History is capped at **10 versions per name** (oldest deleted beyond that). Saving a brand-new name creates no history entry.

`/context-diff` compares versions:

```text
/context-diff ProjectA                      # newest archived version vs. current
/context-diff ProjectA --versions           # list archived timestamps
/context-diff ProjectA 20260101-120000Z     # that version vs. current
```

Output is a plain unified diff (`diff -u`). Friendly messages are printed when the snapshot or its history does not exist.

## Staleness Warning

`/context-load` appends a final WARNING line when the snapshot file's mtime is older than **7 days** (override with the `SESSION_CONTEXT_STALE_DAYS` environment variable), suggesting regeneration via `/context-generate`. File age uses `stat -f %m` (macOS) with a `stat -c %Y` fallback (Linux).

## Snapshot Content

A context snapshot should capture what another session needs to know without reading the whole project:

```text
# Session Context: <name>
Generated: YYYY-MM-DD HH:MM
Project: <current directory>

## What Was Done
## Files Changed
## Key Decisions
## Open Issues
## Where I Left Off
## Notes for Next Session
```

For API-heavy projects, include endpoint contracts, data models, auth requirements, error formats, and conventions.

## Current Implementation

The current implementation supports manual generation (with automatic version archival), listing (with version counts), loading (with staleness warning), diffing, removing, and sharing. The share command sends a tmux message telling the target session which snapshot to load; it relies on the project-local snapshot directory being visible to the recipient.

One useful future enhancement is automatic context attachment in `/dispatch`:

```text
/dispatch ProjectA-executor "Build the payment webhook handler"

1. Sender checks for an existing context snapshot.
2. Dispatch message includes a context marker.
3. Receiver loads the snapshot before processing the task.
```

## Design Decision

Context sharing is split into `session-context` instead of living inside `session-chat`. The workflow still interoperates with `session-chat` for tmux pane naming and notifications, but snapshot generation, listing, loading, diffing, and sharing are a separate installable capability.
